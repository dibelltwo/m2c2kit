## Repository Overview

m2c2kit is a TypeScript monorepo (npm workspaces) for building cognitive assessments that run in web browsers and native mobile app WebViews. It uses Google's **canvaskit-wasm** (Skia) for rendering. Node.js >= 22 is required.

## Common Commands

### Build

```bash
# Full monorepo build (respects dependency order automatically)
npm run build

# Build a single package
npm run build -w @m2c2kit/core

# Build for native WebView embedding (no asset hashing, no service worker)
npm run build-webview -w @m2c2kit/assessments-demo
```

### Test

```bash
# All tests
npm run test

# Single package
npm run test -w @m2c2kit/core

# Single test file (must use env-cmd for NODE_OPTIONS)
npx env-cmd -f .env.jest -- jest --selectProjects @m2c2kit/core packages/core/src/__tests__/MyTest.test.ts

# Single test by name
npx env-cmd -f .env.jest -- jest --selectProjects @m2c2kit/core -t "test name"
```

### Development (watch mode)

```bash
cd packages/<package-name>
npm run serve    # concurrent tsc --watch + rollup --watch
```

### Lint / Format

Prettier runs automatically via Husky pre-commit hook (`lint-staged`). To run manually:

```bash
npx prettier --write .
```

## Build Pipeline & Dependency Order

The build is orchestrated in sequential stages (see `build-concurrent` in root `package.json`):

1. `@m2c2kit/build-helpers` — must build first; used by all others
2. (concurrent) `@m2c2kit/core`, `@m2c2kit/cli`, `@m2c2kit/schema-util`, `@m2c2kit/data-calc`, `@m2c2kit/test-helpers`
3. (concurrent) `@m2c2kit/session`, `@m2c2kit/addons`, `@m2c2kit/physics`, `@m2c2kit/survey`, `@m2c2kit/db`
4. (concurrent) `@m2c2kit/embedding`, `@m2c2kit/schematics`, all assessment packages
5. `@m2c2kit/assessments-demo`, `@m2c2kit/assessments-registry`

When building a single package manually, its upstream dependencies must already be built.

## Architecture

### Session / Activity Pattern

- **`Session`** (in `@m2c2kit/session`) is the top-level container. It manages one or more **activities**.
- **`Activity`** is the interface all runnable units implement. Lifecycle: `initialize → start → [data events] → end/cancel`.
- **`Game`** extends `Activity` for interactive canvas-based assessments.
- **`Survey`** extends `Activity` for survey-js surveys.
- Activities emit typed events: `onStart`, `onData`, `onEnd`, `onCancel`.
- Data schemas are JSON Schema format and auto-generated by `@m2c2kit/schema-util` into each package's `schemas.json`.

### Canvas Rendering

`@m2c2kit/core` wraps **canvaskit-wasm** (v0.40.0). All drawing uses Skia primitives exposed through m2c2kit's `Shape`, `Label`, `Sprite`, `Button`, etc. The WASM binary is bundled with the core package and copied to assets at build time via rollup.

> **Note:** A custom canvaskit-wasm build exists (see `BUILDNOTES.md`) for older Android devices with Adreno GPU text distortion. Not needed normally.

### Embedding in Native Apps

`@m2c2kit/embedding` abstracts platform differences. Call `Embedding.initialize(session, { host: "MobileWebView" })` in the assessment entry point, then the session communicates with the native shell via:

- **Android:** `window.AndroidM2c2` (`@JavascriptInterface` in Kotlin)
- **iOS:** `window.webkit.messageHandlers.iOSM2c2` (`WKScriptMessageHandler` in Swift)

The session is also exposed on `window.m2c2kitSession` for external JS control.

Native shell examples are in `examples/android-simple-webview/` and `examples/ios-simple-webview/`.

### dist vs dist-webview

| Target | Command | Use case |
|--------|---------|----------|
| `dist/` | `npm run build` | Web deployment, hashed assets, service worker |
| `dist-webview/` | `npm run build-webview` | Native iOS/Android WebView, no hashing, no service worker |

### Assessment Package Pattern

Each assessment package (e.g., `@m2c2kit/assessment-color-dots`) follows this structure:

```
src/
  index.ts          ← Game subclass + parameters + scenes
  __tests__/        ← Jest tests
rollup.config.mjs   ← Production bundle
rollup.config.runner.mjs ← Local dev runner
schemas.json        ← Auto-generated parameter schemas
```

Build steps per package: `clean → tsc → rollup → schemas`.

### Creating a New Assessment

```bash
npm install -g @m2c2kit/cli
m2 new myapp
cd myapp
npm run serve
```

The CLI uses Angular DevKit schematics (`@m2c2kit/schematics`) to scaffold from templates. The generated structure mirrors `@m2c2kit/assessment-cli-starter`.

### Testing Infrastructure

- Framework: **Jest** with `ts-jest` (`useESM: true`)
- Environment: `jsdom` for canvas packages, `node` for CLI/build-helpers
- The `.env.jest` file sets `NODE_OPTIONS=--experimental-vm-modules --max-old-space-size=3072`
- `@m2c2kit/test-helpers` provides mock implementations for testing game logic without a real canvas
- `ts-jest-mock-import-meta` mocks `import.meta` for ESM compatibility

### CI/CD

GitHub Actions (`.github/workflows/ci.yml`) on push to `main`:
1. Full build
2. CodeQL analysis
3. Docker-based CLI integration tests (`npm run integration-test`)
4. Publishes updated packages to npm if tests pass
5. Triggers static site and docs deployments
