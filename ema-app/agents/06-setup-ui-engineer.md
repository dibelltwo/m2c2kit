# Agent 06 — Setup UI Engineer

## Role
Build the study configuration and participant setup screens that run inside the Capacitor WebView. This is the researcher/coordinator-facing UI where a study protocol is configured, assessments are selected, and the participant is enrolled. Runs before any EMA assessments begin.

## Owns
```
ema-app/app/src/setup/
  index.html              ← setup screen entry point
  setup.ts                ← main setup logic
  components/
    protocol-form.ts      ← schedule config form
    assessment-picker.ts  ← assessment selector + parameter editor
    enrollment.ts         ← participant ID entry + server enrollment
    preview.ts            ← summary of configured protocol
  styles/
    setup.css
```

## Enrollment Screen

```typescript
async function enrollParticipant(participantId: string, protocol: StudyProtocol) {
  // 1. POST to backend /participants
  const res = await fetch(`${API_BASE}/participants`, {
    method: "POST",
    body: JSON.stringify({ participant_id: participantId, protocol }),
  });
  const { token } = await res.json();

  // 2. Save token + protocol locally (Capacitor Preferences)
  await Preferences.set({ key: "participant_token", value: token });
  await Preferences.set({ key: "study_protocol", value: JSON.stringify(protocol) });

  // 3. Trigger scheduler to generate + register notifications
  sendToNative({ type: "PROTOCOL_SET", protocol });
}
```

## Saved Protocol Access

After enrollment, the setup screen should be locked (PIN or coordinator-only mode). The participant should never see the setup screen during normal study use.

```typescript
// Check if already enrolled on app launch
const protocol = await Preferences.get({ key: "study_protocol" });
if (protocol.value) {
  // Skip setup → go directly to waiting-for-notification state
  showStandbyScreen();
} else {
  showSetupFlow();
}
```

## Design Constraints
- **No external CSS frameworks** — use minimal vanilla CSS to keep bundle small
- **No React/Vue** — plain TypeScript + DOM APIs (setup screen is simple enough)
- **Accessible:** all inputs must have labels, error messages must be announced
- **Coordinator-first:** optimize for speed of setup, not for participant use
- **Validation:** prevent impossible schedules (e.g., more prompts than windows allow given min-gap)

## Integration Points

- **Reads from:** Assessment Engineer's registry (`@m2c2kit/assessments-registry`) for assessment list + schemas
- **Reads from:** Protocol Architect's `study-protocol.schema.json` for validation
- **Sends to:** Backend Engineer's `POST /participants` on enrollment
- **Sends to:** Native Platform (via bridge `PROTOCOL_SET` event) to trigger scheduler
- **Saves locally:** via Capacitor Preferences (protocol + token)

## Does NOT
- Render assessment content (m2c2kit handles that)
- Schedule notifications (Scheduler agent handles that)
- Implement data sync (Data & Sync agent handles that)
- Build the backend enrollment endpoint
